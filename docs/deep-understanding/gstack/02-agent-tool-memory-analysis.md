# 02-agent-tool-memory-analysis.md — gstack

## What Counts as the "Agent Loop" in This Repo

gstack does not implement its own agent loop. Instead, it **extends Claude Code's built-in agent loop** by injecting rich system prompts (SKILL.md files) that instruct Claude how to behave.

The effective agent loop is:
```
Claude Code (external binary)
  ├── Reads SKILL.md into context window
  ├── Decides next step based on SKILL.md instructions
  ├── Calls tools: Bash, Read, Edit, Write, Agent
  │   └── Bash often invokes $B (browse commands) → HTTP → Chromium
  ├── Processes tool output
  └── Repeats until SKILL.md workflow is complete
```

gstack's role is to **configure** this loop (via SKILL.md) and to **provide** the primary tool it wraps (the browse daemon).

## Single-Agent, Multi-Agent, or Hybrid?

**Hybrid: primarily single-agent with optional multi-agent fanout.**

- **Default (single-agent)**: Most skills run in the Claude Code session that the user invoked. The agent reads the skill, executes steps sequentially, calls tools.
- **Multi-agent fanout (optional)**: The `ship` and `autoplan` skills can spawn sub-agents via the Agent tool for parallel review phases. The `review-army.ts` resolver generates a dispatch mechanism that coordinates CEO, Design, Eng, and DX review sub-specialists concurrently.
- **Forced single-agent for security**: The `codex` skill explicitly prevents sub-agent access to `~/.claude/` and `.gstack/` directories — a trust boundary for the external Codex AI.

## Coordinator / Subagent Patterns

### `/autoplan` — Auto-Review Pipeline
```
Coordinator (autoplan skill in Claude Code session)
  ├── Phase 1: CEO Review  ── spawns subagent with ceo-review SKILL.md
  ├── Phase 2: Design Review ── spawns subagent with design-review SKILL.md
  ├── Phase 3: Eng Review  ── spawns subagent with eng-review SKILL.md
  └── Phase 4: DX Review   ── spawns subagent with devex-review SKILL.md
```
- Mandatory sequential execution (not parallel): each phase gates the next
- Decision classification: Mechanical → auto-decide silently; Taste → surface at final gate; User Challenge → never auto-decide

### `/ship` — Shipping Workflow
```
Coordinator (ship skill)
  ├── Pre-ship: run tests, check review status
  ├── Mid-ship: version bump, CHANGELOG, merge
  └── Post-ship: canary verification (spawns canary subagent)
```

### `/review` — Review Army
```
Coordinator (review skill)
  ├── Critical pass (serial)
  └── Specialist dispatch (parallel fanout):
      ├── SQL/Data Safety reviewer
      ├── Race Condition reviewer
      ├── LLM Output Trust reviewer
      ├── Shell Injection reviewer
      └── Enum Completeness reviewer
```

### Test Sessions (E2E)
```
test/helpers/session-runner.ts
  └── Spawns: claude -p <prompt>  (isolated subprocess)
      └── Claude reads test fixture SKILL.md
      └── Executes workflow
      └── NDJSON output piped back → parseNDJSON()
```
This is a distinct multi-agent pattern: the parent test runner spawns Claude as a subordinate process to evaluate behavior, not a subagent in the Claude SDK sense.

## Tool Invocation Model

gstack's skills use three categories of tools:

### 1. Standard Claude Code Tools
- **Bash**: Run shell commands, git operations, npm/bun scripts
- **Read / Edit / Write**: File system operations
- **Agent**: Spawn sub-sessions (used sparingly, for review fanout)

### 2. Browse Tool (custom)
- Invoked via `$B <command>` in Bash
- `$B` is a shell alias or binary path to `browse/dist/browse`
- HTTP POST to daemon at `127.0.0.1:PORT/command`
- Commands: 13 read + 20 write + 17 meta = 50 total
- Refs: `@e1`, `@e2`... — agent references DOM elements by ref, not XPath/CSS

### 3. External AI Tools
- `codex` CLI — invoked via Bash for second-opinion review
- `claude -p` — invoked via session-runner.ts in tests (not during user-facing skills)

### Tool Selection Pattern (in SKILL.md templates)
Skills declare `allowed-tools:` in frontmatter. Example for `ship`:
```yaml
allowed-tools:
  - Bash
  - Read
  - Edit
  - Write
  - Agent
  - TodoWrite
```
Claude's tool selection within a skill is model-discretion unless the SKILL.md template explicitly instructs which tool to use at each step.

## Memory Model

gstack uses **three distinct memory layers**:

### 1. In-Context Memory (ephemeral, session-scoped)
- Everything in the current Claude Code context window
- The SKILL.md prompt is always in context (costs tokens on every turn)
- Preamble instructions (context recovery, checkpoints) help Claude maintain state within long workflows
- **Context recovery preamble**: generated by `generate-context-recovery.ts` — tells Claude how to re-read its own TodoWrite state if context gets pruned

### 2. File-System Memory (persistent, project-scoped)
- `~/.gstack/learnings/` — cross-session learnings log
- `~/.gstack/projects/$SLUG/evals/` — eval history with trend data
- `~/.gstack/config.yaml` — user preferences (explain_level, team_mode)
- `~/.gstack/security/session-state.json` — security state (cross-process)
- Project-local files written by skills: TODOS.md, CHANGELOG.md, VERSION

### 3. External Memory (Supabase, inferred)
- `bin/gstack-telemetry-sync` suggests telemetry is synced to Supabase
- Exact schema unknown without reading supabase/functions/

### Memory Access Pattern in Skills
```
Skill start: read TODOS.md, CHANGELOG.md, VERSION (file-system memory)
During workflow: write TodoWrite (in-context scratchpad)
On completion: update TODOS.md, CHANGELOG.md (persist to file-system)
Cross-session: read ~/.gstack/learnings/ (persistent knowledge base)
```

## Session Model

- **Sessions are user-initiated**: User invokes `/ship` or another skill in Claude Code
- **Sessions are long-running** (minutes to hours for ship/autoplan)
- **No automatic session resumption**: If Claude Code exits, the skill must be re-invoked
- **Context save/restore**: `/context-save` + `/context-restore` skills provide explicit session checkpointing — write a summary to file, load it on resume
- **Server sessions are independent**: The browse daemon runs across Claude Code sessions (persists 30min idle); skill sessions do not

## Context-Passing Model

### Within a skill (single session)
- TodoWrite stores task state in context
- Shell commands output to stdout → parsed by Claude
- File reads bring external state into context
- Browse commands output structured text with @refs

### Between coordinator and subagent
- Agent tool call includes explicit prompt with all necessary context
- Subagent has no access to parent's context window
- Results returned as text from Agent tool
- **Implication**: coordinator must explicitly pass all relevant information in the Agent prompt

### Between skill and next skill invocation
- No automatic carry-over
- `/context-save` + `/context-restore` provide manual bridging
- File-system state (TODOS.md, VERSION, CHANGELOG) implicitly carries state

## Retry / Recovery Model

### Browse daemon
- CLI: one automatic retry on ECONNREFUSED (server crash mid-session)
- Health check: 8-30s polling loop on startup
- Version mismatch: kills stale server, restarts transparently
- Server lock: prevents concurrent startup (atomic `wx` flag)

### Skill-level recovery
- `generate-confusion-protocol.ts`: instructs Claude to stop and ask user when confused (rather than hallucinate progress)
- `generate-context-recovery.ts`: instructs Claude to re-read TodoWrite state if context was pruned
- `generate-continuous-checkpoint.ts`: periodic state writes during long workflows

### Eval-level recovery
- `session-runner.ts`: no retry on claude -p failure (fail fast, report error)
- `llm-judge.ts`: retries on 429 rate limits

## Points of Determinism vs Model Discretion

### Deterministic (enforced by code)
- Canary injection and detection (L5 security) — always fires deterministically
- Token-based authentication — binary pass/fail
- Server port randomization — OS-level
- Diff-based test selection — exact glob matching, not fuzzy
- Atomic server state file writes — OS-level atomicity
- Version mismatch detection — exact binary hash comparison

### Model Discretion (prompt-guided)
- Which skill step to execute next (Claude reads SKILL.md instructions)
- Whether a finding is "critical" or "medium" severity in reviews
- How to phrase AskUserQuestion output
- Whether to fix an issue inline or add to TODOS.md
- Whether a test failure is "pre-existing" (explicitly warned against in CLAUDE.md)
- Sidebar model routing (Opus vs. Sonnet): `pickSidebarModel()` is a regex heuristic, but model interprets ambiguous cases

### Hybrid (heuristic threshold → model interpretation)
- ML classifier scores (0.0-1.0) → threshold rules → verdict (deterministic once score is computed, but score is stochastic)
- `autoplan` decision classification: Mechanical/Taste/User Challenge (model classifies the decision type using a taxonomy from the prompt)

## What Would Break First Under Scale or Ambiguity

### Under Scale
1. **Single Chromium process**: One browser instance per user; no horizontal scaling for concurrent tab operations
2. **In-context memory**: Context window fills with large SKILL.md prompts + history; complex skills (ship: 42K tokens) leave little room for large diffs
3. **Local state files**: `~/.gstack/browse.json`, `~/.gstack/security/session-state.json` — race conditions if multiple Claude Code instances run concurrently on the same machine (only partially mitigated by server lock)
4. **Eval cost**: At ~$4/run (gate) and ~$3.85/run (E2E), scale → cost. No batching or caching of eval results.

### Under Ambiguity
1. **Decision classification in autoplan**: The Mechanical/Taste/User Challenge taxonomy is model-discretion; ambiguous cases may be auto-decided when they should be surfaced
2. **ML classifier FP rate**: Stack Overflow-style instruction writing was a documented false positive source; the ensemble rule mitigates but doesn't eliminate
3. **Confusion protocol vs. forward progress**: In long workflows, Claude may oscillate between asking for clarification and proceeding independently
4. **Multi-host skill divergence**: As host configs get more complex, template placeholders may produce subtly different behavior across hosts (Codex, Factory, OpenCode) that's hard to test
